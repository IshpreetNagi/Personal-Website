FUNLAND ARCADE

See Code

Details

This project embodies my thorough and extensive knowledge and understanding of the language where I started my programming journey, Python. The project is essentially an arcade emulator and is composed of three games of varying variety, each with its own scoring system, sounds, pictures, style, music, start screen, and end screen. The games themselves are a quiz game, a number guessing game, and imitation of the widely popular arcade classic, Whack-a-mole.

Below I expand on the games themselves while also touching upon their inner workings and some challenges I was met with through its construction. To view the source code of the arcade emulator, please visit the Github regarding this project using the button above.


Question interface from the Quiz Show game

Throughout this entire program, your mouse is the main tool used to perform all actions. Fundamentally, the screen of the game is made to be a grid, and your mouse’s position is being tracked on it at all times. The position of your mouse on the grid is stored in three variables, one holding the coordinates on the x-axis, the other on the y-axis, and the third is responsible for holding the clicking action of the mouse. These variables are then used to determine if the mouse is within the invisible hitbox parameters that are created on top of objects such as circles and squares. These invisible hitbox parameters give the objects the characteristics of buttons, thus allowing the user to interact with the program through the use of their mouse click.

Let’s commence with discussing the ‘Quiz-Show’ game. In that game the user is essentially asked 6 questions and are rewarded a point with each question they get correct. This is essentially a multiple choice quiz, as along with each question, the user is greeted with 4 possible answer options. The answer choices are displayed on 4 interactable boxes that are programmed to be buttons, thus allowing the user to simply click on the answer they think is correct with the help of their mouse. Then according to the option the user chose, the respective if or elif-statement will be followed. Selecting the correct answer will trigger an if-statement, that will complete the following actions, adding a value of one to the variable that holds the number of questions you answered correctly, and playing a cheering sound effect to inform the user on their correct decision. However, selecting an incorrect option, will result in the according elif-statement being triggered, thus resulting in nothing being added to the variable that holds the number of questions you got right, therefore showing you got the question wrong, along with the playing of a ‘booing’ sound effect to notify the user that they choose the incorrect answer. With each question, there is a 10-second timer, prompting the user to answer the question within the time frame. Failing to do so will result in the question being marked incorrect and the user losing that point. At the end of the game, the number of questions you answered correctly being held by the variable is displayed on the screen, thereby showing the user their final score out of a total of 6, while also greeting the user with two interactable button options, letting them choose between returning to the main menu to select a new game, or replay the quiz game.

Moving on to the second game, the number guessing game. This program will prompt the user to think of any whole number between 1 and 100, and then will continue to try to guess that number. When presented with the guessed number, the user would select a button telling the program either the guessed number is the number is higher than your number or lower. To start, the program would first automatically generate a random number to just establish a kind of limit, which it would then use to make its following guesses. If the user clicks the button telling the code that it guessed the correct number, then the game essentially ends and you are taken to the end screen. However, if the user is prompted that the guessed number is too high or too low, then the code will learn from that response and adjust the range of numbers it would guess in the future appropriately. For example, if the program first guesses 75 and the user answers that the number is too high, then the program will set its maximum parameters to 75 and will not guess a number higher than 75. If the program then guesses 34 for example, and the user answers that 34 is too low, then the code will set 34 as its minimum number and will only guess numbers higher than 34 but less than 75. Then as it guesses more numbers and the user continues to give it more information about the number they are thinking of, the program will keep adjusting its max and min values appropriately until it finally arrives at the number you are thinking of. This program is essentially a very basic form of a machine learning algorithm, it was more of me trying to enter the A.I world a little to explore its inner workings and all the possibilities it has to offer.


Interface of the Guess-it game

The third and last game in the program is a Whack-a-Mole imitation game called, “Gopher Game”. Much like whack-a-mole, the objective of the game is to catch the gopher by clicking on him when it randomly appears out of one of four possible gopher holes on the ground. Once the gopher appears on the screen, the user has exactly one second to quickly click on the gopher thus catching it, failing to do so will arrive the user at the end screen, where much like the other end screens, the user can then choose to either return to the main menu or reply the gopher game. In order to win, the user would have to successfully click on the gopher 3 times in a row. Essentially the way the game works is that I have created a invisible hitbox for the gopher to be the exact same size as the gopher picture, and whatever random coordinates the gopher picture appears on the screen, at the same coordinates the gopher hitbox would appear, thus giving the gopher the same interactable characteristics as any other interactable buttons found throughout this program. This way when the mouse is clicked, and if the coordinates of the mouse on the screen are within the coordinate parameters of the gopher button, it would register that as a catch. Furthermore the coordinates of where the gopher will appear on screen are not just randomly generated to be so that the gopher could appear anywhere on the screen. The randomized coordinate generator has restrictions to it making it so the gopher would only appear within the positions of the 4 gopher holes, thus actually giving the user an idea of where the gopher would possibly pop-up. The point received upon catching the gopher symbolizes the number of times the user has caught the gopher and is stored in a variable. Once that variable holds a value that is equal to 3, an if-statement is triggered, thus resulting in the game to end and the user to be brought to the winning screen.


Interface of the Gopher Game

The most challenging task I encountered while creating this program was the creation of the hitboxes around the interactable objects. Essentially it was a lot of guess-and-check while creating the hitboxes, I had to first guess the coordinate range of where each of the objects was in relation to the game screen. Once I had discovered that I have to find the area of the object so that I could understand how many coordinates I needed to highlight so that it can actually be turned into an interactable object. After making my assumptions on all the coordinates, I would have to continually run the code and make the appropriate adjustments.
